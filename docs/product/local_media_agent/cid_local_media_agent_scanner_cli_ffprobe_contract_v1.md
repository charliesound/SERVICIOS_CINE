# CID Local Media Agent — Scanner CLI ffprobe Contract v1

## Phase

`CID.LOCAL_MEDIA_AGENT.SCANNER.CLI.FFPROBE.CONTRACT.V1`

## Objective

This phase defines the future `ffprobe` integration contract for the local scanner CLI before any runtime implementation.

The goal is to specify how CID Local Media Agent may inspect local audiovisual files safely while preserving the local-only privacy model.

This phase is contract-only and test-only.

## Current scanner

The current scanner script is:

`scripts/cid_media_agent_scan.py`

## Scope

This contract defines future requirements for:

- optional `ffprobe` availability checks
- safe subprocess execution
- timeout handling
- local-only media inspection
- metadata allowlist
- privacy-safe output
- deterministic failure behavior
- explicit path exposure modes
- human review requirements before editorial use

## Non-goals

This phase does not implement `ffprobe`.

This phase does not call `ffmpeg`.

This phase does not call `ffprobe`.

This phase does not read real media metadata.

This phase does not transcribe media.

This phase does not create subtitles.

This phase does not create proxies.

This phase does not generate DaVinci files.

This phase does not upload media.

This phase does not touch SaaS runtime.

This phase does not touch DB, models, Alembic, Docker, frontend, Stripe, AI Jobs, credits or ledger.

## Local-only rule

All future `ffprobe` execution must happen on the client machine or local workstation.

The scanner must never upload:

- original video files
- original audio files
- proxies
- waveforms
- thumbnails
- transcripts
- subtitles
- extracted frames
- technical metadata dumps containing absolute local paths

## ffprobe availability

The future scanner must treat `ffprobe` as an optional local dependency.

If `ffprobe` is not available, the scanner must:

- continue the safe baseline scan when possible
- record a controlled warning
- avoid crashing with a raw traceback
- avoid leaking absolute media paths
- avoid suggesting cloud upload as a fallback

## Safe subprocess execution

Future `ffprobe` calls must use bounded subprocess execution.

Required controls:

- no shell=True
- explicit argv list
- timeout
- captured stdout and stderr
- controlled stderr truncation
- no environment secrets in output
- no recursive command expansion
- no user-controlled command templates
- no network access requirement

## Timeout behavior

If `ffprobe` times out, the scanner must:

- mark the affected item as `technical_metadata_status=timeout`
- continue scanning other files when safe
- avoid dumping raw command output
- avoid exposing absolute input paths unless explicit opt-in path exposure is enabled

## Metadata allowlist

The scanner may only persist a narrow allowlist of technical metadata.

Allowed future fields may include:

- duration_seconds
- format_name
- codec_type
- video_codec
- audio_codec
- width
- height
- frame_rate
- sample_rate
- channel_count
- timecode_detected
- stream_count
- probe_status
- probe_warning_code

The scanner must not persist full raw `ffprobe` JSON by default.

## Privacy and path policy

The existing scanner path-policy remains authoritative.

Default outputs must not expose local absolute paths.

The default path representation must remain privacy-safe, such as:

- sanitized_path
- local_relative_path when safe
- hashed_path
- redacted_path

`local_absolute_path` may only be emitted with explicit opt-in.

The future `ffprobe` layer must not bypass the current path-policy.

## Output files

Future technical metadata may be written only under the configured local output root.

Allowed future output locations must be inside the local project output tree, such as:

- `01_media_catalog/media_catalog.json`
- `01_media_catalog/media_catalog.csv`
- `01_media_catalog/media_catalog.md`
- `01_media_catalog/technical_metadata.json`
- `99_logs/privacy_events.jsonl`
- `99_logs/scanner.log`

The scanner must not create files outside `--output-root`.

## Raw ffprobe output

Raw `ffprobe` stdout and stderr must not be stored by default.

A future debug mode may store diagnostic information only if:

- explicit opt-in is provided
- the output is local-only
- sensitive path material is redacted
- the debug file is clearly marked as unsafe for sharing
- human review is required before external use

## Failure codes

Future failures must use stable status codes.

Examples:

- `ffprobe_missing`
- `ffprobe_timeout`
- `ffprobe_invalid_json`
- `ffprobe_unsupported_media`
- `ffprobe_permission_denied`
- `ffprobe_probe_failed`
- `ffprobe_privacy_redacted`

## Human review

Technical metadata generated by `ffprobe` must be treated as production assistance, not final truth.

Human review is required before:

- editorial decisions
- delivery assumptions
- conform assumptions
- timecode-based sync decisions
- DaVinci handoff
- external sharing

## Acceptance criteria

This contract is accepted when:

- the contract document exists
- tests verify contract-only scope
- tests verify no runtime `ffprobe` implementation is introduced in this phase
- tests verify local-only privacy language
- tests verify subprocess safety requirements
- tests verify metadata allowlist requirements
- tests verify raw `ffprobe` output is forbidden by default
- tests verify SaaS, DB, Docker, Alembic, frontend, Stripe, AI Jobs, credits and ledger are out of scope
