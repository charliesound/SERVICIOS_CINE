# AILink Sync Dialogue — Trusted Person Feedback v1

## 1. Objetivo

Este documento define cómo enseñar AILink Sync Dialogue a una persona de confianza después de que el dry run interno haya obtenido PASS interno.

La finalidad es recoger feedback real y seguro antes de enseñar la demo a escuelas, productoras, leads fríos o redes sociales.

Esta fase responde directamente a:

- AILINK.PRODUCT.SYNC_DIALOGUE.DEMO.READINESS.PHASE7.
- AILINK.PRODUCT.SYNC_DIALOGUE.DEMO.READINESS.GATING.AUDIT.PHASE7.1.
- AILINK.PRODUCT.SYNC_DIALOGUE.DEMO.READINESS.GATING.EVIDENCE.PHASE7.2.
- AILINK.PRODUCT.SYNC_DIALOGUE.DEMO.READINESS.GATING.EVIDENCE.RUN.PHASE7.3.
- AILINK.PRODUCT.SYNC_DIALOGUE.DEMO.SCRIPT.REFINEMENT.PHASE7.4.
- AILINK.PRODUCT.SYNC_DIALOGUE.DEMO.SCRIPT.DRY.RUN.PHASE7.5.

Esta fase es documental y operativa. No implementa código, no modifica scanner, no modifica matching, no modifica exports, no modifica reportes, no crea UI real, no crea backend, no crea frontend, no crea instalador, no crea n8n, no crea CRM, no toca PostgreSQL real, no toca Docker, no toca runtime, no toca configuración, no graba vídeo público y no modifica CID SaaS.

## 2. Estado de entrada

Estado de entrada:

- Phase7.5 obtuvo PASS interno.
- El ensayo duró entre 5 y 7 minutos.
- Se explicó LIMITED PASS.
- Se explicó beta controlada.
- Se explicó local-first.
- Se explicó que no es sincronización automática final.
- Se explicó el caso 0 match suggestions en e2e dummy.
- Se pidió feedback concreto.
- No se vendió.
- No se cerró precio.
- No se prometió DaVinci, Avid, Premiere, waveform sync, transcripción ni claqueta visual.
- No se abrió material sensible, configuración, CID SaaS, n8n ni CRM.

Estado máximo de salida:

- PASS interno validado con persona de confianza.
- PASS interno con ajustes.
- LIMITED PASS mantenido.
- FAIL de demo externa.

Esta fase no puede producir PASS público.

## 3. Quién puede ser persona de confianza

Puede ser:

- Profesional audiovisual cercano.
- Montador conocido.
- Ayudante de montaje conocido.
- Técnico de sonido cercano.
- Productor de confianza.
- Docente cercano.
- Colaborador técnico que entienda el contexto beta.
- Persona capaz de dar feedback honesto sin tratarlo como lanzamiento público.

No debe ser:

- Lead frío.
- Escuela contactada por primera vez.
- Productora externa sin confianza previa.
- Cliente potencial al que se quiera vender.
- Público de LinkedIn.
- Público de Facebook.
- Público de YouTube.
- Grupo grande.
- Persona que pueda interpretar la demo como producto terminado.

## 7. Estructura recomendada de sesión

Duración total recomendada:

- 10 a 20 minutos.

Bloques:

1. Contexto breve: 1 a 2 minutos.
2. Demo hablada: 5 a 7 minutos.
3. Preguntas de comprensión: 3 a 5 minutos.
4. Feedback abierto: 5 minutos.
5. Cierre y decisión: 1 minuto.

La demo hablada debe seguir la estructura de Phase7.5:

- Apertura.
- Problema.
- Qué hace hoy.
- Evidence run real.
- Lectura del reporte.
- Explicación del e2e dummy.
- Límites honestos.
- Feedback solicitado.
- Cierre.

## 15. Matriz de decisión

### PASS interno validado

Requisitos:

- La persona entiende el problema.
- La persona entiende beta controlada.
- La persona entiende local-first.
- La persona entiende que no es sincronización final.
- La persona entiende el 0 match suggestions del dummy.
- La persona ve valor en el reporte.
- La persona no interpreta que se esté vendiendo un producto final.
- No se detectan frases peligrosas.
- No se abre material sensible.
- No recomienda cambios críticos antes de otra persona de confianza.

Resultado:

- Puede enseñarse a una segunda persona de confianza.
- Todavía no es PASS público.
- Todavía no se publica en LinkedIn.
- Todavía no se abre a escuelas/productoras como campaña.

### PASS interno con ajustes

Causas:

- La persona entiende el valor, pero sugiere mejoras de lenguaje.
- La persona entiende el flujo, pero pide enseñar antes el reporte.
- La explicación de 0 match suggestions necesita simplificación.
- La demo dura demasiado.
- La demo suena demasiado técnica.
- La demo necesita una captura más clara.

Resultado:

- Ajustar guion o material visual.
- Repetir con otra persona de confianza.
- No abrir todavía a escuelas/productoras.

### LIMITED PASS mantenido

Causas:

- La persona no entiende bien el valor.
- La persona cree que se promete sincronización final.
- La persona confunde beta con producto terminado.
- La persona no entiende el reporte.
- La persona no entiende el 0 match suggestions del dummy.
- La persona detecta que faltan evidencias.

Resultado:

- Volver a ajustar demo.
- No enseñar a más personas todavía.

### FAIL

Causas:

- Se abre material sensible.
- Se prometen funciones no implementadas.
- Se intenta vender.
- Se da precio cerrado.
- Se presenta como demo pública.
- La persona interpreta que es producto final.
- La demo daña la credibilidad.

Resultado:

- Detener salida externa.
- Volver a Phase7.4 o Phase7.5.
- No contactar escuelas/productoras.

## 16. Condición para avanzar

Se puede avanzar si:

- El feedback da PASS interno validado o PASS interno con ajustes menores.
- No hay promesas peligrosas.
- No hay confusión crítica.
- La persona entiende beta controlada.
- La persona entiende que no es demo pública.
- Se registran ajustes concretos.

No se puede avanzar si:

- El feedback da LIMITED PASS mantenido o FAIL.
- La persona interpreta que es producto final.
- La persona no entiende el valor.
- La explicación de 0 match suggestions falla.
- Se detecta una promesa peligrosa.
- Se abre material sensible.
