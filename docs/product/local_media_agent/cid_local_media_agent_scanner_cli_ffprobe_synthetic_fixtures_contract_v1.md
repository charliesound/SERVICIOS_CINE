# CID Local Media Agent — Scanner CLI ffprobe Synthetic Fixtures Contract v1

## Phase

CID.LOCAL_MEDIA_AGENT.SCANNER.CLI.FFPROBE.SYNTHETIC.FIXTURES.CONTRACT.V1

## Objective

This phase defines the contract for future synthetic media fixtures that may be used by a later ffprobe synthetic probing implementation.

This phase is documentation-only and test-only.

It does not create media fixtures.

It does not generate video files.

It does not generate audio files.

It does not execute ffprobe.

It does not execute ffmpeg.

It does not modify scanner runtime.

It does not add subprocess execution.

It does not inspect real media.

## Current baseline

The previous phase defined the future ffprobe synthetic probe contract.

This phase narrows the fixture rules before any future implementation creates or probes synthetic assets.

The scanner must remain unchanged in this phase.

## Scope

This contract defines:

- allowed synthetic fixture categories
- forbidden source material
- fixture naming policy
- future fixture folder policy
- maximum size and duration policy
- privacy and metadata requirements
- commit safety requirements
- validation expectations
- human review requirements

## Non-goals

This phase does not implement ffprobe probing.
This phase does not implement fixture generation.
This phase does not add fixture binary files.
This phase does not add real media.
This phase does not call ffprobe.
This phase does not call ffmpeg.
This phase does not add subprocess runtime.
This phase does not modify scripts/cid_media_agent_scan.py.
This phase does not create technical_metadata output.
This phase does not touch SaaS runtime.
This phase does not touch DB, models, Alembic, Docker, frontend, Stripe, AI Jobs, credits or ledger.

## Future fixture location

A later explicit implementation phase may place synthetic fixtures under:

tests/fixtures/local_media_agent/ffprobe_synthetic/

The folder must be created only in that later implementation phase.

This contract phase must not create that folder.

## Allowed future fixture categories

A future implementation may use only these fixture categories:

- synthetic_video_minimal
- synthetic_audio_minimal
- synthetic_video_with_audio_minimal
- synthetic_unsupported_media_placeholder
- synthetic_corrupt_or_invalid_media_placeholder
- synthetic_permission_denied_placeholder

The first future probing implementation should start with the smallest possible set.

## Recommended future filenames

Future fixture filenames should use safe descriptive names:

- synthetic_video_minimal.mp4
- synthetic_audio_minimal.wav
- synthetic_video_with_audio_minimal.mp4
- synthetic_unsupported_media_placeholder.txt
- synthetic_invalid_media_placeholder.bin
- synthetic_permission_denied_placeholder.dat

The names must not include client names, project names, people names, location names, camera roll names, sound roll names, scene identifiers from real projects, production company names, drive names or dates from real shoots.

## Future fixture size and duration policy

Future media fixtures must be tiny.

Recommended limits:

- maximum duration: 2 seconds
- maximum individual file size: 512 KiB
- maximum total fixture folder size: 2 MiB
- minimum resolution sufficient for ffprobe recognition only
- minimum audio duration sufficient for ffprobe recognition only

If a later implementation needs larger fixtures, that must be approved in a separate phase.

## Future generation policy

Future synthetic media fixtures should be generated from deterministic commands or tiny committed assets only if explicitly approved.

If generated by command:

- the generation command must be documented
- the command must not use real source media
- the command must not embed personal metadata
- the command must not embed project metadata
- the command must not depend on external internet access
- the command must be reproducible locally

If committed as files:

- each file must be documented as synthetic
- each file must be small enough for the repository
- each file must be safe to share
- each file must be reviewed before commit
- binary additions must be explicit in the phase scope

## Forbidden fixture sources

Future fixtures must not be derived from:

- real camera originals
- real sound rolls
- real production proxies
- real editorial exports
- real dailies
- real rushes
- real subtitles
- real transcripts
- real screenshots
- real thumbnails
- real production documents
- private pilot material
- client material
- downloaded stock media
- online videos
- copyrighted media samples
- mobile phone clips
- drone clips
- screen recordings from real projects

## Metadata privacy requirements

Future fixtures must not embed:

- personal names
- production company names
- project names
- GPS coordinates
- device serial numbers
- camera serial numbers
- lens serial numbers
- local user names
- home directory paths
- drive labels
- cloud paths
- network paths
- real shoot dates
- real location names
- comments copied from real files

The future probe implementation must not persist raw ffprobe JSON by default.

The future probe implementation must not persist full format tags or full stream tags.

## Future fixture manifest

A later implementation phase should add a fixture manifest if actual fixtures are created.

Suggested future manifest path:

tests/fixtures/local_media_agent/ffprobe_synthetic/fixture_manifest.json

The future manifest should include:

- fixture_id
- relative_path
- fixture_category
- synthetic_origin
- expected_probe_status
- expected_media_kind
- expected_has_video
- expected_has_audio
- expected_duration_policy
- expected_privacy_safe
- human_reviewed

The manifest must use relative paths only.

The manifest must not include absolute paths.

## Future expected statuses

Future fixture tests may cover these statuses:

- available
- unsupported_media
- invalid_json
- permission_denied
- probe_failed
- privacy_redacted

The missing ffprobe status is already covered by availability preflight tests and must not require a media fixture.

Timeout tests may be simulated with a mocked subprocess layer in a later phase and must not require large media files.

## Future path policy

Future fixture outputs must respect the scanner path-policy.

Default output must use sanitized paths.

local_relative_path may be enabled explicitly.

hashed_path may be enabled explicitly.

redacted_path may be enabled explicitly.

local_absolute_path remains explicit opt-in only and must not be used in shareable fixture reports.

## Future output policy

Future fixture probing outputs must remain under --output-root.

Allowed future output files:

- 01_media_catalog/media_catalog.json
- 01_media_catalog/technical_metadata.json
- 00_project/processing_status.json
- 99_logs/warnings.json
- 99_logs/errors.json
- 99_logs/privacy_events.json

No output may be written beside the fixture source files.

No fixture source file may be modified.

## Future test policy

A later implementation phase must test:

- fixture manifest validity
- fixture file existence
- fixture file size limits
- relative path only storage
- no absolute path leaks
- no raw stdout persistence
- no raw stderr persistence
- no raw ffprobe JSON persistence
- allowed metadata schema only
- no scanner output outside --output-root
- no source fixture modification
- human review flag present

## Human review

Fixtures are synthetic, but human review is still required before they become accepted test assets.

Human review must verify:

- the fixture is not real production material
- the fixture contains no private metadata
- the fixture is small enough
- the fixture is safe to commit if binary
- the fixture has a documented purpose
- the fixture does not create product claims

## Real media remains blocked

Real media probing remains blocked after this contract.

A future real media phase must be explicit and separate.

The project must first pass synthetic fixture contract, synthetic probe contract and synthetic implementation validation before considering any real media probing.

## Acceptance criteria

This contract is accepted when tests verify:

- the phase name is present
- the phase is documentation-only and test-only
- no fixtures are created in this phase
- no scanner runtime is modified
- no subprocess runtime is added
- no ffprobe or ffmpeg execution is added
- allowed fixture categories are documented
- forbidden source material is documented
- fixture naming policy is documented
- future fixture folder policy is documented
- future manifest policy is documented
- metadata privacy restrictions are documented
- output restrictions are documented
- real media remains blocked
